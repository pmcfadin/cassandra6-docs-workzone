Why Cassandra

Built for the Hardest Problems

Cassandra was purpose-built at Facebook for inbox search — the kind of workload where failure is not an option and scale is non-negotiable. That DNA shows.

◎

Distributed by Design

Every node is equal. No master, no single points of failure. Data is distributed and replicated across the cluster transparently.

◈

Fault Tolerant

Replication across multiple data centers keeps your application running through hardware failures, network partitions, and full datacenter outages.

⟁

High Performance

Cassandra consistently outperforms alternatives at scale. Writes in particular are extremely fast — the log-structured storage engine is optimized for throughput.

⊞

Linearly Scalable

Add capacity by adding nodes. Throughput scales linearly with cluster size. Some of the largest Cassandra deployments run thousands of nodes.

⟳

Elastic Operations

Expand or contract the cluster without downtime. Cassandra rebalances data automatically as you add or remove nodes.

⊛

Enterprise Security

TLS encryption in transit, authentication, authorization, and auditing. Node-to-node encryption and client-to-node encryption are configurable independently.

Project structure

How Cassandra Is Run

Apache Cassandra is a top-level project of the Apache Software Foundation (ASF). The project is overseen by a Project Management Committee (PMC) elected by existing committers. All binding project decisions are made through consensus on the public dev@ mailing list.

Releases follow the ASF release policy. All release artifacts are voted on publicly. The voting thread, signing keys, and checksums are available for every release. Cassandra versioning follows a major.minor.patch scheme with documented upgrade paths between versions.

Cassandra 6.0
Released

Vector search, extended LWT, and unified compaction bring the project's most significant feature release in four years.

Apache Cassandra 6.0 marks a significant step forward for one of the most widely deployed distributed databases in production infrastructure. The release introduces native vector search capabilities, making Cassandra a compelling platform for applications that combine operational and AI workloads without requiring a separate vector store.

The release also delivers extended support for lightweight transactions across a broader set of query patterns, reducing the operational complexity of applications that require conditional writes at scale. Unified compaction, a strategy long in development, consolidates previous strategies under a single adaptive engine that responds to workload characteristics at runtime.

Upgrading from 5.0 follows a standard rolling upgrade path. Users on 4.1 LTS should review the migration guide for schema and driver compatibility notes before beginning the upgrade sequence.

Highlights in 6.0
  • Native vector search with ANN indexing via SAI integration
  • Extended lightweight transactions (eLWT) across complex query patterns
  • Unified compaction strategy replaces STCS, LCS, and TWCS tuning
  • Storage-attached indexes promoted to stable for production use
  • Read repair modernization reduces background coordinator load
  • Java 21 virtual thread support in the driver and server runtime

How to Contribute

  1. Start with the docs or JIRA. Good first contributions are often documentation improvements, test coverage, or triage help on open issues.
  2. Subscribe to the dev list. Project decisions happen on the mailing list. Reading the archive gives context before writing code.
  3. Read the contributing guide. Covers code style, commit messages, ticket etiquette, and the review process before you open a patch.
  4. Open a JIRA ticket for any non-trivial change before writing code. Aligns the work with project direction and avoids wasted effort.
  5. Submit a GitHub pull request against apache/cassandra. Tag the JIRA ticket and request a review on the dev list.

Governance is handled by the Project Management Committee (PMC). New committers are nominated by existing committers and voted in by the PMC. The project operates under the Apache Way.

Architectural Properties

Distributed
Every node in a Cassandra cluster is identical. There is no single point of failure, no primary node, and no coordination bottleneck for writes or reads.
Fault Tolerant
Data is automatically replicated across multiple nodes and optionally across data centers. Node failures are transparent to applications and handled without operator intervention.
Performant
Writes and reads are served from memory-resident structures before being persisted. Benchmark consistently shows sub-millisecond p99 latency at sustained throughput.
Scalable
Capacity scales linearly by adding nodes. A cluster can grow from three nodes to hundreds without schema changes, downtime, or rebalancing operations.
Elastic
Nodes join and leave the cluster transparently. Virtual nodes distribute data automatically, making expansion and contraction operationally straightforward.
Secure
Client-to-node and node-to-node TLS encryption, pluggable authentication, role-based authorization, and auditable access controls are available out of the box.
Governance

Apache Cassandra is governed by a Project Management Committee (PMC) under the Apache Software Foundation. The PMC oversees the release process, committer nominations, and project direction. Development decisions are made by consensus on the public mailing list. All committers and PMC members are listed on the ASF phonebook. Cassandra Enhancement Proposals (CEPs) are the mechanism for significant changes to the project; all accepted and in-progress CEPs are tracked in the CEP index.
